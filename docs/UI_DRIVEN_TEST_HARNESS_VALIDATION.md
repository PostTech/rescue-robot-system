# Test UI 기반 검증 하네스 아키텍처 정밀 타당성 보고서 (UI_DRIVEN_TEST_HARNESS_VALIDATION.md)

## 1. 개요

본 보고서는 비즈니스 로직의 올바름을 검증하기 위해 **'Test UI(테스트 벤치)와 E2E 시나리오 테스트 하네스(Playwright/Cypress 등)를 연동하는 UI-Driven Test Harness'** 아키텍처의 타당성을 평가한다.

현재 이 프로젝트의 `src/web/` 하위에 구현되어 있는 **웹 관제 대시보드 프론트엔드 실물**을 기준으로 이 모델의 압도적인 엔지니어링적 장점과 동시에, **실제 로봇 제어 및 분산 비동기 시스템에서 마주하는 치명적인 현실적 한계와 비판점**을 가감 없이 고찰한다.

---

## 2. 현재 시스템의 프론트엔드 실물 결합 구조

이 프로젝트는 단순 텍스트나 API에 그치지 않고, 실제로 관제 및 로직 시각화를 위한 완벽한 웹 대시보드 환경을 보유하고 있다.

- `src/web/index.html`: 관제 메인 레이아웃 및 3D 지형 렌더링 뷰포트 배치.
- `src/web/js/app.js` & `websocket.js`: 백엔드 REST API와 실시간 WebSocket 채널을 통해 로봇 제어 상태, 지형 분석 데이터, 이벤트 수신 처리.
- `src/web/js/terrain_viz.js` & `detection_viz.js`: 3D LiDAR 기반 지형 격자 데이터 및 센서 퓨전 요구조자 검출 결과를 캔버스 상에 실시간 시각화.

따라서 사용자께서 제안하신 **"Test UI 및 자동 인터랙션 하네스"의 대상물이 이미 소스 코드 상에 확실하게 존재함**을 확인하였다.

---

## 3. UI-Driven 테스트 하네스의 본질적 가치 (장점)

1. **블랙박스 아키텍처의 투명화 (End-to-End Truth)**:
   - 백엔드의 험지 주행 정책(`search_drive_policy.py`)이나 비동기 센서 퓨전 엔진(`decision_engine.py`)의 수학적 무결성이, 실제 관제 화면 상에서 위험 경보 적색 팝업으로 사용자에게 잘 보이는지를 검증하는 E2E 테스트는 기획서 명세를 보장하는 가장 확실한 물리적 입증 수단이다.
2. **AI 자가 치유 피드백 루프의 가속화**:
   - 가상 브라우저가 화면의 "임무 승인" 버튼을 눌렀을 때 렌더링에 실패하면, 하네스가 즉각 브라우저 콘솔 오류 로그와 스크린샷 데이터를 3단계 구현 AI에게 재주입하여 코드를 자가 복구할 수 있는 직관적인 피드백 소스를 제공한다.

## 4. [실랄한 비판] UI-Driven 테스트 하네스의 3대 현실적 한계와 취약점

사용자께서 제시해주신 이상적인 아키텍처 모델에도 불구하고, 실제 임베디드 로봇 제어 시스템과 분산 웹 환경에서는 다음과 같은 **혹독한 한계점과 부작용**이 존재한다.

```text
  [ 브라우저 (Playwright) ]  ── UI 인터랙션 ──>  [ Vanilla JS (Direct DOM) ]
           │                                             │
      (비결정적 지연)                              (상태 오염/Ghost DOM)
           │                                             │
           ▼                                             ▼
  [ API / WebSocket ]       ── 비동기 버스 ──>   [ 로봇 시뮬레이터 (Mock) ]
                                                   (물리 렌더링 지연/데드락)
  
  ☞ 결과: Flaky Tests 지옥 (동일 코드에서 성공/실패가 무작위 반복)
```

### 한계 1: Flaky Tests의 지옥 (비동기 타이밍 & Race Condition)
- 로봇 시스템은 실시간 분산 이벤트 버스(`event_bus.py`)와 비동기 네트워킹을 기반으로 동작한다.
- Playwright 가상 유저가 "임무 승인 버튼"을 클릭한 직후, 백엔드 지형 분류 프로세스와 로봇 SLAM 상태 동기화가 이루어져 UI 화면에 "승인 완료" 메시지가 렌더링되기까지는 찰나의 네트워크 지연(Network Latency)과 스레드 스케줄링 갭이 존재한다.
- E2E 하네스가 이 타이밍 대기(`waitForTimeout`)를 밀리초 단위로 정교하게 통제하지 못하면, **시스템 코드는 완벽함에도 네트워크 렉으로 인해 테스트가 불규칙하게 실패하는 Flaky Test 현상**이 상시 발생한다. 이는 AI 에이전트에게 가짜 오류 로그를 전송하여 불필요한 코드 파괴 재시도를 유발하는 주범이 된다.

### 한계 2: Vanilla JS의 Direct DOM 조작 구조 취약성
- 현재 `src/web/js/app.js`는 React나 Vue 같은 엄격한 리액티브 상태 관리 프레임워크가 아닌, **Vanilla JS 환경에서 DOM 엘리먼트를 직접 Select하여 값을 업데이트하는 방식**을 취하고 있다.
- 이러한 구조에서는 데이터의 흐름과 화면 렌더링 간에 '단일 진실 원천(Single Source of Truth)'이 깨지기 쉽다. UI 테스트 하네스가 DOM 텍스트(`document.getElementById("status").innerText`)를 읽는 시점에, 이전 상태의 잔상(Ghost DOM)을 읽어와 무수한 Assertion False Alarm을 일으키는 구조적 한계가 있다.

### 한계 3: 실물 시뮬레이터(ROS/WebRTC) 가상화 비용의 한계
- UI 상에서 로봇 조이스틱 버튼을 눌렀을 때 로봇이 장애물을 잘 피해 주행하는지를 진짜로 E2E 검증하려면, 브라우저뿐만 아니라 백엔드 하부의 3D 로봇 물리 엔진(Gazebo 등)과 ROS2 노드를 통째로 헤드리스(Headless) CI 환경에 띄워야 한다.
- 이는 하네스 셋업 및 메인테넌스 비용을 기하급수적으로 폭증시키며, 단지 가벼운 비즈니스 로직 테스트를 수행하기 위해 지나치게 무거운 인프라 족쇄를 채우는 **'주객전도(Over-Engineering)'**가 된다.

## 5. 최적화 제언: 하이브리드 검증 하네스 모델 (Hybrid Harness Model)

우리는 위에서 고발한 혹독한 현실적 한계들을 회피하면서 UI 검증의 이점을 취하기 위해 다음과 같은 **'하이브리드 검증 하네스'**를 구축할 것을 강력히 권고한다.

```text
┌─────────────────────────────────────────────────────────────────┐
│ 1. 뷰모델 (ViewModel) 오프라인 격리 검증 (1차 방어선 - Light)     │
│    - 브라우저 없이 UI 상태 관리 로직(View Model)만 메모리 검증    │
│    - 예: tests/unit/ui/test_mission_setup_view_model.py         │
└────────────────────────────────┬────────────────────────────────┘
                                 ▼
┌─────────────────────────────────────────────────────────────────┐
│ 2. 라이트웨이트 E2E 시각적 검증 (2차 방어선 - Heavy)             │
│    - Playwright/Cypress는 오직 Mock API 데이터 바인딩 상태에서   │
│      최종 컴포넌트 렌더링 정합성 및 화면 텍스트 유무만 미니멀 검증│
└─────────────────────────────────────────────────────────────────┘
```

1. **1차 방어선: UI 뷰모델(ViewModel) 오프라인 격리 검증**:
   - 현재 본 프로젝트는 [test_mission_setup_view_model.py](file:///c:/Users/cosmo/AI_challange/tests/unit/ui/test_mission_setup_view_model.py)와 같이 브라우저 없이도 UI의 상태 상태와 로직만을 빠르게 단독 검증하는 유닛 하네스를 완벽하게 띠고 있다.
   - 브라우저를 띄우지 않아 속도가 100배 빠르며 타이밍 이슈(Flaky)가 전무하므로, 비즈니스 로직의 올바름을 검증하는 1차 방어선으로 극대화하여 활용해야 한다.
2. **2차 방어선: 라이트웨이트 E2E 검증**:
   - 실물 ROS/DB 통신을 전부 Mock 인터페이스로 전치한 고정된 테스트 샌드박스 위에서만 Playwright 자동 스크립트를 가동하여, 오직 "버튼 클릭 시 모달 렌더링 여부" 등 최소한의 렌더링 계약 조건만 엄격히 assertions한다.

## 6. 결론

사용자께서 지적하신 **"최종 검증은 사용자가 바라보는 화면(UI)의 정상 동작을 테스트 하네스로 묶어 증명해야 한다"**는 주장은 소프트웨어 E2E 품질 확보의 완벽한 정답이다. 

그러나 분산 제어 환경에서는 타이밍 지연과 Vanilla JS 상태 관리의 한계로 인해 E2E 하네스가 스스로 무너져 내리는 **'Flaky Test의 늪'**에 빠지기 쉽다. 

따라서 우리는 이미 구축되어 있는 **'오프라인 UI 뷰모델 테스트 하네스'를 뼈대로 삼아 로직의 결함을 1차적으로 초고속 예방**하고, 브라우저 E2E 하네스는 고정된 API 샌드박스 위에서 최소한의 렌더링 규격만 체크하는 **하이브리드 전략**을 취해야만 비결정론적 AI가 아키텍처를 안전하게 보존하도록 통제할 수 있다.
